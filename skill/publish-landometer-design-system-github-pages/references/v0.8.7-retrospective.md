## สิ่งที่พลาดก่อนจะได้วิธีนี้

### 1. คิดว่า archive เป็นทางออกของ binary/connector limitation

การส่ง ZIP/TAR ให้ผู้ใช้อัปโหลดเองไม่ได้แก้ release tree และ GitHub Pages ไม่แตกไฟล์ให้อัตโนมัติ วิธีนี้เพิ่ม checksum และ extraction risk โดยไม่จำเป็น.

**กฎถาวร:** เขียน final source tree เข้า GitHub โดยตรง. ถ้ามี binary ใหม่ ให้ใช้ direct base64 Git blob/tree/commit และตรวจ exact hash; ห้ามซ่อนใน archive.

### 2. สรุปเร็วว่า connector เขียน repository ไม่ได้

GitHub connector มี `create_file`, `update_file`, `create_blob`, `create_tree`, `create_commit`, `update_ref`, PR และ merge actions แต่ต้องค้น schema ที่ตรงก่อน.

**กฎถาวร:** สำรวจ connector write actions ให้ครบก่อนบอกว่าติดข้อจำกัด.

### 3. ใช้ scratch branch ทดลองจน diff ปนไฟล์ชั่วคราว

Branch ทดลองมี `release-sources`, chunks และ temporary workflow จำนวนมาก จึงไม่ควรแก้ต่อหรือใช้เป็น PR head.

**กฎถาวร:** เมื่อ branch polluted ให้สร้าง clean branch จาก `main` แล้วเขียนเฉพาะ frozen allowlist.

### 4. เท่ากับ repository state = live state

การเห็น v0.8.7 บน `main` ยังไม่พิสูจน์ว่า Pages deploy แล้วหรือ CDN เสิร์ฟไฟล์ใหม่.

**กฎถาวร:** release เสร็จเมื่อ live HTML/manifest/assets และ browser behavior ผ่านการตรวจ endpoint จริงด้วย retry/cache bypass.

### 5. ใช้ QA marker เพื่อ trigger workflow

Marker PR ใช้กระตุ้น workflow ได้ แต่ marker ไม่ใช่ production source และไม่ควร merge.

**กฎถาวร:** เก็บ reusable workflow; ปิด QA-trigger-only PR โดยไม่ merge marker.

### 6. เสี่ยงอ้าง `machineValidation: passed` เกินหลักฐาน

Repository validator และ package alignment ไม่ได้ปิด Thai, recognition, identity/media, screen-reader และ artifact-level gates โดยอัตโนมัติ.

**กฎถาวร:** คงสถานะจริง เช่น `source_limited`, `noindex`, `machineValidation: pending` จน applicable gates ผ่านครบ.
